# Coding Rules

## 读取与验证
- 编辑代码前，先读取目标仓库的 README、package manifest 或等效入口文件。
- 不要在阅读相关文件或运行最小有效检查之前解释代码行为。
- 不要仅从文件名推断框架 hook、配置源或数据流。
- 实现后，说明验证状态和残留风险。

## 外科手术式修改
- 每行改动都必须能追溯到用户请求；改不到的就不要改。
- 不要"顺手"重构、改格式、改注释、调引号、加 type hint。
- 匹配现有代码风格，即使你认为有更好写法。
- 看到无关死代码先说出来，不要直接删除。
- 只清理本次改动产生的孤儿 import/变量/函数。

## 目标驱动
- 多步任务先把"做某事"翻译成可验证目标，再开工。
  - "加校验" → "为无效输入写测试，让它通过"
  - "修 bug" → "写能复现的测试，再让它通过"
- 多步实现前列简明计划：`[步骤] → 验证：[检查]`。

## 记忆
- 如果发现可能再次重要的跟进事实或仓库特定坑，记录到 `context/projects/`。
- 变更保持在任务范围内，保留现有项目约定。
